# 服务压测与业务稳定性测试方案及报告

为了保证服务在真实生产环境以及各种突发流量、海量数据场景下的高可用性，制定本测试与压测方案。


## 1. 大数据与极端场景压测矩阵 (重点)

下述场景专门针对排行榜在**海量数据**与**运行异常**两类极端情况下的稳定性表现进行验证：

### 1.1 海量单榜单压测 (Massive Item Volume)
- **数据规模**：持续打入数据，直至单个排行榜内累计达到 **10万 -> 50万 -> 100万** 个参赛条目（Item）。
- **流量模型**：混合读写（80% 写入数据更新 `POST /leaderboard/{id}/item`，20% 读取 TopN 数据 `GET /leaderboard/{id}?n=100`）。
- **验收准则 (SLO)**：
  - 非 2xx 错误率控制在 **0.5%** 以内。
  - TP99 (p99) 延迟在三个数量级跃升时不应出现超过 30%的性能断崖式下降。
  - 核心内存（RSS）呈现平缓增长，未出现 OOM 被系统 kill 掉的现象。

### 1.2 冷启动与缓存并发击穿验证 (Stampede Protection)
- **测试前提**：主动清空单服内存榜单 map，但保留 Redis 元数据（冷启动场景）。
- **流量模型**：使用 1000 个并发线程瞬间以极其密集的密度访问**同一个**未命中的排行榜。
- **验收准则 (SLO)**：
  - Redis 元数据 `Restore` 逻辑被精准保护，单一时间窗内**仅触发 1 次**重算编译操作。
  - 单次卡顿处理完后，后续 999 次请求直接从内存得到实例，成功率达到 **99.5%** 以上。

### 1.3 调榜热点（帕累托）与海量调度重算测试
- **测试前提 1**：创建 **1000** 个不同的排行榜，模拟 20% 的热门榜单占用了 80% 的查询带宽。检查长尾小榜单会不会因此资源饿死（可用性测试）。
- **测试前提 2**：设置某排行榜为 `scheduled` 延迟调度模式，并积累 **10万+** 的待处理 `dirty item`（脏数据）。
- **流量模型**：在极高脏数据积累下，外部主动触发一轮 `Recompute` 重算操作，同时继续接受前台业务写入。
- **验收准则 (SLO)**：
  - 能够分批 (Batch) 成功完成 10w 余条的规则引擎重载。
  - 最后获取的排行榜能够确保表现为严格的积分单调递减（Monotonic Score Fall）。
  - 没有多余或遗漏的脏标记。

### 1.4 底层依赖网络故障混沌测试 (Fault Injections)
- **测试流程**：
  1. 向 Redis 集群注入 **200ms - 500ms** 的瞬时网络拨动（高延迟）。
  2. 人为主动 kill 一小段时间下 Redis 主节点（短时无响应/Timeouts）。
- **验收准则 (SLO)**：
  - 请求将按预期的错误码返回（500，带结构化日志方便排查），而非发生内存雪崩或协程池全部卡死泄漏。
  - 底层恢复连通后，系统能够自我恢复，无须手动重启 Pods。

### 1.5 长效浸泡疲劳测试 (Long-run Soak Test)
- **测试流程**：开启混合查询写入流量并保持满载运行状态至少 **1小时**。
- **验收准则 (SLO)**：
  - 不存在稳定单调上升的内存泄漏（Memory Leak）的曲线。
  - Go runtime GC 的 pause(暂停时长) 控制良好，不存在长时间 Stop-The-World。
  - Goroutine 总量在完成一个波次后能稳步回落。

## 2. 业务正确性验证抽样方案 (Data Correctness)

排行榜不能仅保证“跑得快”，还要绝对保证“算得准”：
1. 大负荷打流量后，随机发出 Top-N 取值（如 N=100，N=500）。
2. 在该采样面内，将返回的最终 `Score` 抽查出至少 1000 组独立数据（包括当时该项原始数据 payload 状态）。
3. 设计离线跑批角本，根据相同 Expression 完全外部复算，对比验证。
4. **通过标准**：
   - Offline 计算评分与 API 取得的评分**差值对比的违和率应 < 0.1%**。
   - 所有在同一时间窗内的排行切片，不允许出现排名错位情况（违反单调性：即排名靠前的分数反而小于后者）。
